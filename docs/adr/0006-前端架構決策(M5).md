# ADR 0006:前端(M5)— 路由/狀態、錯誤處理、401 refresh、搶購輪詢與倒數校準

日期:2026-07-14|狀態:已採納(M5 完成)

## 背景

M5 實作 Vue 3 前端:登入/註冊、活動列表、活動詳情/搶票(倒數 + 輪詢)、訂單列表/詳情/模擬支付、admin 後台(活動/票種 CRUD、預熱、對帳)。設計文件第 1、4、9、10 節與 CLAUDE.md 定了大方向(單一 SPA、Element Plus、baseURL 相對 `/api/v1`、ID 一律 string、倒數用 server time、輪詢指數退避),多處實作細節就地決策。後端(M0–M4)不在本里程碑修改,前端純作為 API 契約的整合對象。

技術基線沿用 ADR 0001:Vite + Vue 3.5 + vue-router 5 + Pinia 3 + Element Plus 全量引入 + TypeScript(oxlint + ESLint,`ci:lint` 無 `--fix`)。

---

## 1. 分層與模組邊界

- **`api/`**:純資料層。`http.ts`(axios instance + 攔截器)、`types.ts`(對照後端 DTO 的契約型別)、各領域 API(`auth/events/seckill/orders/admin`)。
- **`stores/`**:Pinia(僅 `auth`)。**token 事實來源不放 store**,改置於無依賴的 `api/tokenStorage.ts`,讓 `http.ts` 攔截器與 store 都能存取而不循環 import(§3)。
- **`composables/`**:`useSeckillFlow`(搶購流程封裝,§5)。
- **`utils/`**:`datetime`(UTC 顯示 + 倒數格式化)、`serverClock`(全站後端時鐘,§6)、`eventCache`(訂單頁把 ID 解析為可讀名稱)。
- **`views/`**:頁面元件;admin 頁在 `views/admin/`。

**取捨**:未上 Vuex-style 集中 store,狀態盡量留在元件內(`ref`);唯一跨頁狀態(登入)才用 Pinia。理由:MVP 頁面間耦合低,過早集中化增加樣板。

## 2. 錯誤處理:攔截器統一 + 業務碼客製雙軌

- **統一軌**:`http.ts` response 攔截器解析 `{code,message,data}`;`code≠0` 一律轉 `BizError(code,message,httpStatus)` 並 `reject`,預設以 `ElMessage.error(message)` 彈全域提示。網路層錯誤(無 response)提示「網路連線異常」。
- **客製軌**:對「需要頁面就地處理」的呼叫,API 層帶 `silent:true`(擴充 `AxiosRequestConfig`),攔截器**不**彈訊息,由呼叫端依 `BizError.code` 客製(如登入 1002 顯示在表單、搶購 3004/3005/3006/3007 各自友善提示)。
- **理由**:多數錯誤「彈一下就好」用統一軌零樣板;少數需就地語意的用 silent 精準控制。避免「全域彈窗 + 頁面提示」重複出現。
- **落實 CLAUDE.md**:controller 不吞例外、統一格式——前端對稱地在**單一攔截器**解析,不在每個 `.catch` 重複解析 `code`。

## 3. token 儲存與 401 refresh 重放(防迴圈)

- **儲存**:`tokenStorage.ts` 以 `ref` + `localStorage` 持久化 access/refresh。獨立於 store 與 http,打破 `http.ts ↔ stores/auth.ts` 的潛在循環 import。
- **判定用業務碼 1401 而非 HTTP 401**:憑證失效由後端回 `code=1401`(ADR 0002)。攔截器只在 `code===1401` 觸發 refresh;登入失敗 `1002`、refresh 失效 `1005` 天然不進重放路徑,不會誤觸迴圈。
- **single-flight**:多個請求同時 1401 時,共用同一個 `refreshPromise`,只發一次 `/auth/refresh`;refresh 用**原生 axios**(非本 instance)呼叫,避免 refresh 自身的回應再進攔截器。
- **防無限迴圈**:每個請求帶 `_retried` 旗標,重放一次為限;`_retried` 已設或無 refreshToken 或 refresh 失敗 → 清 token 並 `window.location.assign('/login?redirect=…')`。
- **取捨**:refresh 失敗採**整頁導向**而非 SPA 路由,順帶重置所有前端狀態(最簡單且最不易殘留半登入狀態);代價是一次整頁重載。

## 4. 路由與守衛

- **路由表**:`/`→`/events`;公開 `/events`、`/events/:id`;需登入 `/orders*`;需 ADMIN `/admin/*`;`/login`;萬用 404。除登入/列表外一律 lazy import(route-level code splitting)。
- **守衛**:`beforeEach` 先 `auth.ensureUser()`(有 token 但 user 未載入時補打 `/auth/me`,確保重整後角色可判斷)→ `requiresAuth` 未登入導 `/login?redirect=fullPath` → `role:'ADMIN'` 非 admin 擋回 `/events`(未登入則導登入)→ 已登入者訪 `/login` 導回 `/events`。
- **兩道防線**:前端守衛只是體驗層;admin 真正的安全由**後端 URL 層 + `@PreAuthorize`**(ADR 0003)保證,USER 呼叫 admin API 回 1403。前端守衛減少無效請求,不作為授權依據。
- **redirect 防護**:登入後只接受以 `/` 開頭且非 `//` 的站內路徑,避免 open redirect。

## 5. 搶購流程與輪詢策略(`useSeckillFlow`)

- **流程**:領 token(60s TTL、用後即焚)→ 立刻 `purchase` → 拿 `requestId` 輪詢 `GET /seckill/result/{requestId}`。三支 API 皆 `silent`,錯誤由流程客製。
- **輪詢參數**:初始 1s、退避因子 1.5、單次間隔上限 5s、最多 12 次(總計約 50s)。**只在 QUEUING 續輪;SUCCESS/FAIL 立即停;次數用盡回 `TIMEOUT`**。單次輪詢網路失敗視同 QUEUING 計次續輪。元件卸載即 `cancelled`,停止輪詢。
- **理由**:1s 起步符合設計文件第 2 節;指數退避 + 12 次上限確保「絕不無限輪詢、不打爆後端」;50s 上限遠小於後端 `result` key 的 10 分鐘 TTL,不會因前端先放棄而錯過已寫入的結果。
- **`TIMEOUT` 不視為失敗**:訊息已入列,後端仍可能成單;引導使用者到「我的訂單」確認,而非報錯。呼應 ADR 0004「DLQ 路徑不寫 result key → 前端輪詢逾時」的已知限制——逾時是預期的收斂行為而非錯誤。
- **防狂點**:搶購期間所有搶購鈕 `disabled`、當前鈕 `loading`,並顯示不可關閉的「排隊中」dialog。
- **業務碼提示**:3004 限流「請求過於頻繁」、3005 售罄(重載詳情)、3006 重複購買(引導看訂單)、3007 憑證失效(請重按)、其餘顯示後端訊息並重載。

## 6. 倒數一律用後端 server time 校準(`serverClock`)

- **決策**:`GET /events/{id}` 回 `serverTime`;載入時 `calibrate(serverTime)` 算 `offsetMs = serverTime − 本機時間`,之後所有倒數用 `serverNow() = Date.now() + offsetMs`,**不信任瀏覽器時鐘**(CLAUDE.md / 設計文件第 6 節)。
- **共用**:`serverClock` 為全站單例,活動詳情校準後,訂單頁的付款期限倒數也共用同一 offset(避免各頁各自校準)。
- **票種四態**:未開放(OFFLINE)/ 即將開賣(距開賣倒數,鈕禁用)/ 開賣中(距結束倒數,鈕啟用;remaining=0 顯示售罄)/ 已結束。**跨界自動重載**:任一票種相位變化時重取詳情(更新即時剩餘並重新校準),使用者無需手動刷新即見「即將開賣→開賣中」。
- **取捨**:offset 只在載入詳情時校準一次,長時間停留頁面若本機時鐘漂移不再修正;對搶票場景(停留數分鐘)足夠,且跨界重載會順帶重新校準。

## 7. ID 與型別:全程 string

- 所有 Snowflake ID(userId/eventId/ticketTypeId/orderId/requestId)在 `types.ts` 一律 `string`,**絕不 `parseInt`/`Number`**(超過 JS `2^53` 會精度丟失,ADR 0002/0005 慣例)。
- **例外**:`price` 顯示用 `Number(price).toLocaleString()`——price 非 ID、僅用於顯示格式化,安全。
- **admin 建票種的 `eventId` 以 string 傳給後端 `Long` 欄位**:Jackson 預設允許 string→Long coercion,實測可行(避免前端組 request 時把 string ID 轉 number)。

## 8. Element Plus 全量引入(沿用 ADR 0001)

- 沿用 `app.use(ElementPlus)` 全量引入;`build` 有單一 chunk >500KB 警告(gzip 約 341KB)。**MVP 接受**,對齊 ADR 0001「M5 再評估 on-demand import」的結論——實測體積可接受,未阻塞驗收;未來優化可換 `unplugin-vue-components` 按需引入。

## 9. 驗收與已知限制

**驗收**:`pnpm ci:lint` / `type-check` / `build` 全綠;手動 E2E(真後端 + 中介軟體)全通:註冊→自動登入→瀏覽活動→(admin)建活動/票種→發布→預熱→(USER)搶購→排隊 dialog→SUCCESS→訂單詳情→模擬支付→PAID;逾時訂單自動轉 EXPIRED 且庫存回補後 reconcile 三方一致;重複購買 3006、售罄禁用、admin CRUD/預熱冪等/對帳一致、USER 呼叫 admin 被擋(1403)皆驗證。

**已知限制**:
- **admin 無訂單管理**:後端無 admin 訂單端點(設計文件第 9 節僅 events/ticket-types/warmup/reconcile);訂單為使用者自助。若需後台訂單管理須擴充後端(超出 M5)。
- **搶購「中途售罄」(3005)競態**:詳情顯示尚有票但 Redis 已扣完的瞬間,只能靠程式碼路徑保證提示與重載,UI 難以穩定重現此時序。
- **倒數 offset 單次校準**:長時間停留且本機時鐘漂移時不再修正(跨界重載會順帶重校),搶票場景足夠。
- **Element Plus 全量 bundle** >500KB(§8),未按需引入。
- **401 refresh 的多請求併發重放**已用 single-flight + `_retried` 防迴圈,但缺自動化測試覆蓋(手動驗證);屬前端純邏輯,後續可加 Vitest。
- **DLQ 導致的輪詢 TIMEOUT**:訊息進 DLQ 時 result key 永為 QUEUING(ADR 0004 已知),前端輪詢 50s 後回 TIMEOUT 引導看訂單——屬需人工介入的異常收斂。

**建議 review 重點**:
- 401 refresh 的 single-flight + `_retried` 防迴圈與整頁導向取捨(§3)。
- 錯誤處理雙軌(統一 vs silent 客製)的邊界(§2)。
- 搶購輪詢參數與 `TIMEOUT` 語意(§5)。
- server time 校準的單例共用與跨界重載(§6)。
- ID 全程 string、`eventId` string→Long coercion(§7)。
