# ADR 0007:監控全套(M6)— stack 佈局、dev/prod scrape 差異、被動告警與閾值

日期:2026-07-16|狀態:已採納(M6 完成)

## 背景

M6 將監控 stack 接起來:Prometheus 抓取、node/postgres/redis exporter、Grafana 四個 dashboard、告警規則。埋點(業務指標、JVM)於 M3 已完成(見 ADR 0004 §9),本里程碑主要動 `infra/` 與少量後端**設定**,不改業務碼。設計文件第 11 節為權威規格(指標表、六類告警、四個 dashboard)。多處落地細節需就地決策,逐項記錄於下。

**範圍縮減(使用者決策)**:本階段只做**被動觀察**用的 dashboard 與告警,供後續壓測時觀察服務與基礎設施;**不接任何主動通知平台(Telegram / email 皆略過)**。因此不引入 Alertmanager,告警由 Prometheus 評估、於 `/alerts` 與 Grafana 被動觀察。

---

## 1. 監控 stack 佈局(決策點 A)

- **決策**:新增 `infra/docker-compose.monitoring.yml`(Prometheus + Grafana + node/postgres/redis exporter),掛到 dev 中介軟體所建立的 **external network `seckill-dev_default`**,使 exporter 以容器名直連 PG / Redis / RabbitMQ。
- **理由**:與 dev 中介軟體同網路,exporter 免額外埠映射即可連線;監控與中介軟體可獨立起停。版本一律 pin(比照 ADR 0001),Prometheus / Grafana 走 named volume 持久化。
- **取捨**:prod 全服務組裝(caddy / frontend / backend + 全套監控於同一 compose)屬 **M7**;M6 只確保「本機能把監控 stack 跑起來並驗證」。exporter 未對主機開埠(僅 Prometheus :9090、Grafana :3000 對 127.0.0.1),縮小攻擊面。

## 2. dev/prod scrape 目標差異(單一設定檔)

- **問題**:backend 在 **dev 跑於本機主機**(非容器),在 **prod 為 Docker 網路內的容器服務**;Prometheus 在兩環境都要抓到它。
- **決策**:`prometheus.yml` 統一以 `backend:8080` 為目標,**不分檔**:
  - **dev**:給 prometheus 容器 `extra_hosts: ["backend:host-gateway"]`,使 `backend` 解析到 Docker 主機 → 本機後端 :8080。
  - **prod**:與 backend 同一 compose 網路,Docker DNS 直接把 `backend` 解析到 backend 容器。
- **理由**:單一設定檔兩環境通用,免維護兩份 scrape 目標;實測 dev 端 `backend` 已解析到 host-gateway(172.x/192.168.65.254)。
- **取捨**:`extra_hosts` 這行只在 dev 的 monitoring compose;prod compose(M7)不需要。相較「dev 用 `host.docker.internal`、prod 用服務名」的分檔做法,本做法讓 scrape 設定零差異。

## 3. node-exporter dev 簡化

- **決策**:dev 的 node-exporter 僅唯讀掛載 `/`(`-/:/host:ro` + `--path.rootfs=/host`),**不加** `pid: host` 與 `rslave` 傳播。
- **理由**:Docker Desktop(Windows)不支援 `rslave`(報 `path / is mounted on / but it is not a shared or slave mount`)。dev 只需 exporter「up 且有樣本」。
- **取捨**:dev 觀察的是 Docker Desktop 的 Linux VM 而非 Windows 主機;prod(OCI Linux)在 M7 prod compose 補上 `pid: host` + `rslave` 取得真實主機視角。

## 4. RabbitMQ per-queue 指標(`return_per_object_metrics`)

- **決策**:新增 `infra/rabbitmq/rabbitmq.conf` 開 `prometheus.return_per_object_metrics = true`,15692 的 `rabbitmq_queue_*` 帶 `queue` 標籤。
- **理由(關鍵,防誤報)**:
  1. **backlog 告警只鎖 `seckill.order.queue`**,排除 `order.timeout.queue`——後者堆積的是 M4 合法的 15 分鐘支付逾時 timer,壓測時本就大量,若用全域聚合會誤觸積壓告警。
  2. **DLQ(`seckill.order.dlq`)可單獨偵測**「出現任何訊息」。
- **取捨**:per-object 指標序列較多(本系統佇列少,成本可忽略)。dashboard 仍用 `sum(...)`,聚合 / per-object 兩模式相容。

## 5. 告警閾值來源與被動模式

- **決策**:9 條規則置於 `infra/prometheus/alert-rules.yml`,閾值**直接取自設計文件第 11 節**六類;`prometheus.yml` 以 `rule_files` 載入;**不設 Alertmanager**。
- **理由**:被動觀察足以支撐壓測需求;規則狀態(inactive / pending / firing)於 `/alerts` 與 Grafana 即時可見,無需通知通道。
- **取捨**:不會主動通知;若日後要 Telegram / email,`prometheus.yml` 已保留 `alerting:` 註解位置,再補 Alertmanager 服務即可。閾值(3000 QPS、p99 500ms、消費併發等)皆待 M7 壓測回頭校準。

## 6. 「消費失敗率」的簡化

- **決策**:設計原文「訂單消費失敗率 > 1%」改以兩條可靠規則表達:**`OrderDlqNotEmpty`(DLQ > 0,critical)** + **`OrderConsumeFailing`(rejected 死信速率 > 0,warning)**。
- **理由**:現有 RabbitMQ 指標**無乾淨的「總消費數」分母**可算百分比;而 `nack(requeue=false)` → DLX 的 `rejected` 死信速率是明確的消費失敗訊號(`order.timeout.queue` 走 expiry 死信,不計入 rejected,不干擾)。DLQ > 0 則是需人工介入的最終訊號(ADR 0004 §7)。
- **取捨**:失去精確百分比語意;換得無誤差、可直接排查的訊號。runbook 已載明此近似。

## 7. 開啟 http histogram 供 API p99

- **決策**:`backend/application.yml` 加 `management.metrics.distribution.percentiles-histogram.http.server.requests: true`。
- **理由**:`http.server.requests` 預設只有 count/sum/max,**無 bucket**,無法 `histogram_quantile` 算 p99;`ApiP99LatencyHigh` 告警需要 bucket。此為純設定變更、不動 Java。
- **取捨**:每個 uri × status 多出一組 bucket 序列(數量可控)。無測試引用 http 指標,CI 不受影響。替代方案 `http_server_requests_seconds_max`(近似,無需此設定)備而不用。

## 8. Grafana provisioning 與祕密

- **決策**:datasource(固定 `uid: prometheus`)與四個 dashboard(`seckill-overview/system/middleware/jvm`)全走 provisioning 自動載入;Grafana admin 密碼、PG/Redis exporter 連線皆走 env(`.env` 填值、`.env.example` 佔位,`.env` 已在 .gitignore)。
- **取捨**:dev 開匿名 Viewer(僅綁 127.0.0.1)方便觀察;**prod 不開匿名**,留 M7。

## 9. Virtual thread 指標缺口

- **現況**:Micrometer 內建 `JvmThreadMetrics` 只涵蓋平台執行緒,**不含 virtual thread 存活計數**(JDK 無便宜的即時 gauge)。JVM dashboard 目前以平台執行緒面板 + 說明面板呈現。
- **決策**:**暫不補**。要真實 vthread 指標(建立速率 `jvm_threads_virtual_started`、pinned 時間)需引入 `io.micrometer:micrometer-java21` 的 `VirtualThreadMetrics` binder(+1 依賴);依 CLAUDE.md 加依賴需確認,使用者本階段未要求,故留待日後。

## 驗收與已知限制

**驗收**(本機實測):Prometheus Targets 6/6 up;`seckill_*` / JVM / node / pg / redis / rabbitmq 指標皆抓到;Grafana 四個 dashboard provisioning 自動載入且有資料(以真實搶購流量灌:success/sold_out/duplicate/invalid_token);9 條告警規則 health=ok;手動 `docker stop` exporter 觸發 `TargetDown` pending→firing→(復原後)resolved;`docs/runbook.md` 完成。

**已知限制**:
- dev node-exporter 觀察 Docker Desktop VM 而非 Windows 主機(§3);prod 才是真實主機。
- 告警閾值為設計初值,待 M7 壓測校準(§5)。
- 「消費失敗率」為近似(§6);virtual thread 即時計數缺口(§9)。
- 被動模式無主動通知(使用者決策);需人工看 `/alerts` 或 dashboard。

**建議 review 重點**:dev/prod scrape 統一做法(§2)、backlog 告警排除延遲佇列的正確性(§4)、消費失敗近似是否可接受(§6)、後端開 http histogram 的取捨(§7)。
